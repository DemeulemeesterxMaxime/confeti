# Leçons apprises

| Date       | Ce qui a mal tourné                                                                                                                                       | Règle pour l'éviter                                                                                                                                                                                                                                   |
| ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 2026-06-26 | Une règle CSS _scopée_ (`<style>` Astro) ne stylait pas un bouton créé via `document.createElement` (margin ignorée)                                      | Tout élément généré en JS doit être stylé dans `<style is:global>` (ancré sur un conteneur, ex. `.ed-grid`), car Astro n'ajoute son attribut de scope qu'au HTML statique du build.                                                                   |
| 2026-06-26 | Preview headless : `getComputedStyle` figé sur la valeur de départ d'une transition (`max-width`) car aucune peinture (rAF ne se déclenche pas)           | Pour lire la valeur CSS _cible_, neutraliser la transition avant mesure (`el.style.transition='none'; void el.offsetWidth;`) ou vérifier la règle, jamais la valeur animée.                                                                           |
| 2026-06-26 | `@astrojs/vercel` 10/11 exige Astro 6/7 (peer dep) ; les XSS d'Astro ne sont corrigés qu'en Astro 6+ : impossible de tout assainir en restant sur Astro 5 | Vérifier le peer range (`npm view <pkg> peerDependencies`) avant un bump. Tracer les advisories non-corrigeables dans `audit-ci.jsonc` plutôt que de baisser le seuil global d'audit.                                                                 |
| 2026-06-26 | Le runtime Vercel utilise Node 22, pas la version locale (24) : risque de divergence CI/prod                                                              | Aligner CI (`.github/workflows`) et `.nvmrc` sur la version de la cible de déploiement (Node 22 pour Vercel), pas sur la version locale du poste.                                                                                                     |
| 2026-06-26 | Un `high` transitif (path-to-regexp via @vercel/routing-utils, épinglé exactement en 6.1.0) bloquait l'audit sans correctif amont non-majeur              | Forcer la version patchée via `overrides` ciblé par version (`"path-to-regexp@6.1.0": "6.3.0"`) plutôt que l'override imbriqué (qui crée un état `invalid`) ; supprimer le lock et réinstaller, puis vérifier la version réelle sur disque + l'audit. |
| 2026-06-26 | Doute sur la lecture des secrets `import.meta.env` côté serveur après la migration Astro 6 (« valeurs toujours inlinées »)                                | Les variables non-`PUBLIC_` restent lues au runtime en SSR : vérifié en lançant `astro dev` avec des env factices et un POST sur l'endpoint (500 fetch failed ≠ 503 non configuré). Tester le runtime plutôt que supposer d'après le changelog.       |
| 2026-07-13 | La navigation gardait tous ses liens desktop sur iPhone, créant un débordement horizontal | Chaque composant de navigation doit prévoir un breakpoint mobile qui conserve les actions prioritaires et réduit les liens secondaires |
| 2026-07-13 | La barre d’actions de l’éditeur restait sur une ligne et élargissait le viewport mobile à 528 px | Vérifier aussi les barres d’outils flexibles sur les plus petits viewports et les autoriser à passer sur plusieurs lignes |
